### **Project Specification: Tactical Matrix & Matchup Counter Engine**

###### **Summary**

The **Tactical Matrix & Matchup Counter Engine** is a high-performance Command Line Interface (CLI) application built from scratch in Python. Its core objective is to act as a data-driven evaluation engine that calculates optimal combinations, tactical counters, and synergy metrics based on structural logic constraints.

Instead of hardcoding game rules or relying on a traditional database wrapper, this engine completely decouples **data (rules)** from **execution (logic)**. It loads external logic rules via structural configuration files (JSON) and builds real-time tactical solutions using pure, predictable algorithmic math and Object-Oriented Programming (OOP).

###### **Why I'm Building This**

As a software engineer, building this specific architecture addresses three critical engineering concepts:

- **Avoiding the "CRUD/Wrapper" Trap:** Many personal projects are just basic entry forms where an external database or an AI API does the heavy lifting. This project forces your Python code to perform the core computations, sorting, and multi-layered data cross-referencing.
- **Mastering Data Decoupling:** In production software, we should never hardcode rules directly into your logic functions. If a game patch changes a character's stats or a playstyle's counter, we shouldn't have to rewrite our Python source code. We simply update the JSON rulebook file, and the engine adapts automatically.
- **Applying Low-Level Logic to High-Level Code:** This bridges the modular, structured, structural-matching mindset learned in low-level memory systems (like our C Virtual Machine) and applies it to clean, reusable Object-Oriented Python structures.

###### **Core Architecture & Mental Model**

The engine operates as a clean, unidirectional data pipeline split into three distinct layers:

```
[ Data Layer: JSON ] ──(Loads Rules)──> [ Engine Layer: OOP Python ] ──(Formats Output)──> [ Presentation Layer: CLI ]
```

###### The Three Layers Explained:

- **The Configuration Data Layer (`tactics.json`):** A raw text file defining the rules of engagement. It maps out entities (e.g., specific playstyles or characters), their baseline attributes, their structural synergies (who they pair well with), and their explicit vulnerabilities (who counters them)

- **The Algorithmic Engine Layer (OOP Python):** The brains of the application. It reads the JSON file, instantiates them into strong Python Objects using reusable classes, and processes user selections using custom sorting and conditional matching algorithms.

- **The Interactive Presentation Layer (CLI Terminal):** A terminal-based execution loop. It takes user commands (like setting up a mockup opponent configuration), passes it to the engine, and outputs beautifully structured text-based reports directly inside your development shell window.

###### **Feature Requirements (What the Code Must Do**

To fulfil the 20-40 hrs portfolio requirement, the system must be built out in three core execution phases:

1. **PI: Entity Modeling & Dynamic Loading**
   - **Dynamic Configuration Parser:** A dedicated file loader module that opens `tactics.json`, validates its structural keys, and dynamically imports the matrix data.

   - **OOP Blueprint Classes:** Implementation of explicit Python classes (e.g., `TacticalEntity`, `MatchupMatrix`) to cleanly wrap attributes, preventing string-parsing spaghetti across the main files.

2. **P2: The Algorithmic Evaluator**
   - **Synergy Metric Calculator:** Loops through a user’s selected draft or configuration and calculates a mathematical cohesion score based on structural synergy tags.
   - **Constraint Matchup Sorter:** When an opponent setup is inputted, the algorithm loops over the matrix, finds every entity that targets the enemy's vulnerabilities, scores them based on efficiency, and returns an ordered list of response strategies from highest to lowest impact.

3. **Phase 3: Interactive CLI Workspace**
   - **Command Execution Loop:** A clean, continuously running `while` shell interface loop that processes custom console text arguments.
   - **Structured Presenter:** A module dedicated to formatting raw terminal text. It will print clean ascii-based layout boards, advantage indicator grids, and clear tactical breakdowns so that data is scannable at a glance.
